# Token Integration Idea

## 0) Intégration passive actuelle : la banque planétaire

Point de départ : chaque planète/token possède une banque collective de son token natif.

Cette banque sert à :
- gate la guerre officielle ;
- débloquer des paliers d'expansion ;
- peser sur les rewards de fin de saison.

Pourquoi c'est passif :
- le token est principalement stocké/déposé ;
- il influence la macro ;
- les holders ont encore peu d'interactions actives directes en gameplay.

## 1) Planet Actions financées par le token natif

Le token natif ne sert pas à acheter du premium individuel, mais à financer des actions collectives temporaires :
- shield de planète ;
- radar ;
- bonus défensif temporaire ;
- accélération d'un chantier collectif ;
- activation d'une opération spéciale.

Objectif :
- donner un rôle communautaire actif au token ;
- rendre l'impact des holders visible dans le monde de jeu.

## 2) Wonders / projets communautaires financés par le token

Idée initiale : financer des grands projets communautaires via token natif.

Exemples :
- Orbital Cannon ;
- Trade Nexus ;
- Spy Network ;
- monument collectif.

Évolution importante :
- les wonders de fin de saison sont déjà construites avec des ressources de base (logique type classic browser strategy) ;
- faire payer directement en « vrai argent » serait trop frictionnel.

Reformulation retenue :
- le token sponsorise l'activité autour des wonders (leaderboard, campagne, bonus d'event), sans remplacer leur coût de base.

## 3) Gouvernance / votes liés au token natif

Couche de gouvernance légère :
- voter une priorité de planète ;
- choisir une cible ;
- prioriser un projet ;
- soutenir une décision diplomatique ;
- activer une campagne communautaire.

Principe :
- pas une DAO lourde ;
- pas un « 1 token = pouvoir total » ;
- plutôt une preuve d'engagement et un poids de coordination.

## 4) Prestige / statut / reconnaissance sociale

Usage du token pour du statut visible :
- titre spécial ;
- bannière ;
- monument des top contributeurs ;
- skin/distinction de prestige.

Note :
- la couche cosmétique est déjà prévue plus tard (skins par planète/token), donc pas prioritaire immédiatement.

## 5) Sponsored Bounties / missions sponsorisées

Idée forte : une team token ou sa communauté dépose son token pour financer des missions/campagnes in-game.

Exemples de campagnes :
- JUP Defense Week ;
- BONK Expansion Campaign ;
- JITO Recon Operation.

Objectifs trackés :
- raids ;
- colonisations ;
- défense ;
- contribution à la planète ;
- production ;
- activité globale.

Valeur :
- le token ne remplace pas les Shards ;
- il crée des événements et incentives communautaires ;
- la team a une raison concrète de pousser le jeu.

## 6) Planet Sponsorship Layer

Variation :
- le token est mobilisé par la team/communauté organisée (pas nécessairement par chaque joueur individuellement) ;
- il sponsorise un event de saison, une offensive, une semaine de défense, un challenge, une campagne de recrutement.

Le token devient une couche de sponsoring communautaire autour du gameplay existant.

## 7) Signal / War Readiness / jauges actives

Version plus gameplay de la banque planétaire :
- War Readiness ;
- Signal ;
- Intel Network ;
- Emergency Reserve ;
- Planetary Morale.

Les dépôts alimentent des jauges consommables pour déclencher des effets/initiatives planétaires.

## 8) Economy perks pour les holders

Avantages modérés envisagés :
- réduction légère du coût de transport ;
- meilleure efficacité commerciale ;
- petit bonus collectif de rendement ;
- perks sur certaines contributions.

Statut actuel :
- partiellement déjà couvert par les bonus holders existants (accélération production, accélération troupes, etc.).

## 9) Règle de design : ne pas remplacer les Shards

Contraintes structurantes :
- le token natif ne remplace pas les Shards ;
- les holders ont déjà des bonus ;
- la cosmétique est prévue ;
- les wonders restent basées sur ressources de base.

Conséquence :
- le token natif n'est pas une premium currency bis.

## 10) Token-funded missions / campaigns

Première piste pleinement compatible :
- la team/communauté finance une campagne in-game en token ;
- la campagne génère objectifs et rewards dédiés ;
- les holders sont incités à agir ensemble.

Exemples :
- campagne d'expansion ;
- campagne de défense ;
- campagne de contribution wonder ;
- campagne de guerre.

## 11) Sponsoring autour des systèmes existants

Principe :
- ne pas remplacer les systèmes core ;
- sponsoriser l'activité autour de ces systèmes.

Exemple wonder :
- la wonder continue de coûter bois/pierre/fer ;
- le token finance une campagne de mobilisation, un leaderboard spécial ou une bonus reward.

Applicable aussi à :
- guerres ;
- défense ;
- colonisation ;
- opérations de planète.

## 12) War / event prize pools sponsorisés

Le token peut financer :
- rewards de guerre ;
- rewards d'événements ;
- pools ponctuels ;
- incentives défense/conquête.

Le token devient un moteur de prize pools communautaires sans collision avec les monnaies/systèmes de base.

## 13) Droit de lancer des opérations communautaires

Le token sert à lancer officiellement une opération communautaire :
- campagne de recrutement ;
- focus sur planète ennemie ;
- prime sur un objectif ;
- mobilisation ;
- campagne de contribution.

Le token = monnaie d'initiative collective.

## 14) Contribution politique / diplomatique

Rôle de coordination politique :
- soutenir une proposition ;
- financer/soutenir une campagne ;
- peser sur une orientation diplomatique ;
- prioriser un axe de planète.

But : renforcer la coordination sans remplacer les autres ressources.

## 15) Principe central

Conclusion-clé :
- le token natif n'est pas une monnaie premium bis ;
- c'est une monnaie d'activation communautaire.

## 16) Objection centrale

Question critique :
- si les teams doivent engager de l'argent/token, quel est leur intérêt concret ?

## 17) Soft ROI : engagement holder mesurable

Valeur possible pour les teams :
- nombre de holders actifs ;
- fréquence de connexion ;
- participation aux campagnes ;
- rétention ;
- usage token in-game.

Coinage peut devenir un outil d'activation communautaire mesurable.

## 18) Soft ROI : branding / utilité du token

Bénéfice narratif/marketing :
- « our token has native utility in a live MMO » ;
- « our holders are the strongest faction » ;
- « our community is active in a real strategy game ».

Pas du cash direct, mais une utilité perçue forte.

## 19) Soft/Hard ROI : acquisition et rétention

Coinage peut servir de canal pour :
- recruter de nouveaux holders ;
- réactiver d'anciens holders ;
- augmenter le temps passé dans l'écosystème ;
- renforcer la fidélité communautaire.

## 20) Hard ROI : revenue share / attribution

Piste plus rationnelle pour les teams :
- attribution de l'activité des holders d'une planète ;
- une partie de la valeur créée peut alimenter :
  - la planète ;
  - un budget de campagne ;
  - un treasury communautaire ;
  - éventuellement la team (selon modèle).

Logique : plus la communauté joue, plus son écosystème gagne.

## 21) Matching par Coinage

Effet levier :
- une team engage du token ;
- Coinage match avec des Shards, de la visibilité, du prize support ou une mise en avant d'event/campagne.

La team n'assume pas seule l'effort.

## 22) Lock au lieu de spend

Mécanique clé :
- au lieu de dépenser, la team lock des tokens pendant X temps ;
- en échange, elle débloque campagne officielle, visibilité, event, statut, reward pool augmenté ;
- à l'expiration, les tokens reviennent (ou le lock se termine).

Modèle souvent plus acceptable qu'un spend pur.

## 23) Utility qui réduit la sell pressure

Si le jeu apporte une utilité de hold/lock :
- plus de stickiness holder ;
- moins de churn ;
- plus de loyauté ;
- plus de raisons de conserver le token.

C'est un argument stratégique même sans revenu direct immédiat.

## 24) Coinage comme couche d'activation prête à l'emploi

Promesse business :
- au lieu de développer un mini-jeu, un système communautaire et des compétitions en interne,
- une team utilise Coinage comme produit gamifié prêt à l'emploi pour activer sa communauté.

---

## Ce qui est déjà intégré / prévu

- bonus holders (accélération ressources, troupes, etc.) ;
- cosmétique spécifique par token/planète (plus tard) ;
- wonders endgame construites avec ressources de base ;
- contrainte forte : ne pas remplacer les Shards.

## Ce qui overlap trop avec les Shards

- token natif comme premium currency bis ;
- token pour speed-up individuels ;
- token pour achat direct de ressources de base ;
- token pour remplacer les coûts core des wonders ;
- economy perks trop proches des bonus holders déjà existants.

## Les 6 pistes prioritaires à creuser

1. Missions/campagnes sponsorisées par token.
2. Sponsoring des systèmes existants (sans les remplacer).
3. Prize pools de guerre/défense/events.
4. Lancement d'opérations communautaires via token.
5. Lock model pour les teams (plutôt que spend).
6. Revenue share / attribution / matching pour créer un upside concret.

## Conclusion finale

Le meilleur positionnement du token natif est :

**outil d'activation communautaire + sponsoring + coordination + campaigns**.

Ce positionnement évite la collision avec :
- les Shards ;
- les ressources de base ;
- les wonders ;
- les bonus holders déjà existants.

Et c'est aussi la proposition la plus défendable pour convaincre les teams de pousser le jeu.
